# 視覺驅動 / YOLO 構想筆記

> 狀態：**構想階段，尚未實作**
> 建立日期：2026-06-02

把「視覺判斷」加進 GreenKeeper 的規劃。記錄決策與技術路線，之後實作時照這份走。

## 目標

讓除草機從「已知整張地圖、走預設路線」進化成 **視覺驅動**：
只靠**前方有限視野**內看到的資訊（哪裡有草、哪裡是障礙），即時決定怎麼走。

這比現在的抽象格子覆蓋更貼近真實割草機 / 掃地機器人的處境。

## 決策：不另開專案

視覺驅動如果跑在模擬器裡，依然是**純瀏覽器**的東西
（onnxruntime-web 或 TensorFlow.js 都能在前端跑 YOLO），GitHub Pages 照樣部署得了。
另開專案會把一個連貫的東西硬拆兩半，所以 **留在同一個 repo**。

## 關鍵技術現實

現在 `sim.js` 的草和除草機都是**抽象格子**（綠色方塊），不是照片。
直接套一個用真實照片訓練的預訓練 YOLO 去看這些方塊，**它什麼都偵測不到**。
所以「視覺驅動」有三條路線：

### 路線 A — 真神經網路 YOLO（學 YOLO 用）
- 把 canvas 畫得更擬真（雜草貼圖／材質）。
- 用自己渲染的畫面訓練一個小 YOLO。
- 前端用 onnxruntime-web 推論。
- 最像「真 YOLO」，但等於訓練一個模型，本身就是一份子專案的工作量。
- 成本：數十 MB 模型權重 + 訓練資料集 + 較重的前端 bundle。

### 路線 B — 模擬感測器模型（研究演算法用）✅ 建議先做
- 不跑真神經網路，而是寫一個「視覺感測器」抽象。
- 除草機只能看到**前方有限視野（FOV）**內的格子。
- 加上偵測雜訊／誤判率，模擬真實感測器的不完美。
- 根據「看到的雜草」決定走法。
- 這是覆蓋演算法研究真正有價值的部分：從「已知整張地圖」變成「只靠局部觀測」。
- 純 JS、零依賴，跟現有架構完美接軌。

### 路線 C — 混合（最終形態）
- 先用路線 B 把「視覺驅動的決策迴圈」架好。
- 之後把感測器後端換成路線 A 的真 YOLO，**介面不變**。

## 建議的實作架構

1. 新增 `vision.js`（感測器／偵測層），跟 `sim.js` 裡的 `ALGORITHMS` 平行。
2. 定義一個介面：

   ```
   sensor(sim) → 偵測到的雜草位置[]
   ```

3. 新增一個演算法 step 函式（例如 `stepVisionDriven`）來消費這個介面。
4. 想玩真 YOLO 時，路線 C 只要替換 `sensor()` 的後端，上層演算法不用動。

**核心好處**：演算法層跟「視覺後端是假的還是真 YOLO」完全解耦，隨時能升級而不重寫。
