# 地圖機制指南（從「瞎走」到梯形分解）

> 建立日期：2026-06-12
> 白話版 user guide：車怎麼知道「我在哪、走過哪」，最終跑上模擬器的 BCD。

## 為什麼需要這個

現在的車是**反應式**：看到人就停，其他時候直直走——它不知道自己在哪、走過哪。
梯形分解（BCD）要的是「把院子分區、規劃路線走好走滿」，前提是**地圖 + 自己的位置**。
這份指南就是補這個前提的路線圖。

## 三個名詞，白話一次講清

| 名詞 | 白話 | 比喻 |
|---|---|---|
| **里程計**（odometry） | 「我走了多遠、轉了多少」的累計 | 汽車儀表板的里程數 |
| **航位推算**（dead reckoning） | 沒有任何感測器，用「我下了什麼指令、跑了多久」去**猜**自己在哪 | 閉著眼睛走路，心裡默數步數 |
| **編碼器**（encoder） | 數輪子實際轉了幾圈的感測器 = 比較準的里程計 | 在輪子上綁計步器 |

兩者關係：**航位推算是「猜」的里程計（會飄），編碼器是「量」的里程計（較準）。**
程式架構一樣，差別只在「走了多遠」這個數字是猜的還是量的。

## 進度條（現在位置 → 目標）

```
[✅ 反應式亂走] → [① 航位推算] → [② 編碼器] → [③ 畫地圖] → [④ 梯形分解 BCD]
                    純軟體          要採購        軟體         移植 sim.js
```

## ① 航位推算（純軟體，現在就能做）

**做什麼**：`auto_drive.py` 每送一個指令（前進/轉彎）就記下時間，事後用
「前進 1 秒 ≈ 走 X 公分、轉彎 1 秒 ≈ 轉 Y 度」回推軌跡，畫成一張俯視足跡圖。

**要先做一次「校正」**（很簡單，拿尺）：
1. 讓車直走 3 秒，量它實際走了幾公分 → 得到「每秒幾公分」
2. 讓車原地轉 3 秒，看它轉了幾圈 → 得到「每秒幾度」

**產出**：`map.png` —— 真車「以為自己走過的路」俯視圖。

**限制（要有心理準備）**：誤差會**累積**。輪子打滑、左右輪不對稱、電池電壓變化
都會讓「猜」越來越偏，跑越久圖越歪。這不是 bug，是這個方法的天性——
所以它是練習與過渡，正解是②。

## ② 編碼器里程計（要採購，百元內）

**採購**：蝦皮搜「**LM393 測速感測器**」（又叫槽型/對射光電感測器），**買 2 個**。
特徵：板上有 ㄇ 字形黑色元件，編碼盤會從凹槽中間穿過。

**已有的**：編碼盤 ×2（零件盒那兩片有縫塑膠盤）✅

**裝在哪**：
- TT 馬達**屁股後面有一根突出的小軸** → 編碼盤套上去
- ㄇ 字感測器騎在盤的邊緣（縫從凹槽中間過）
- 每個感測器 3 條線接 Uno（經 Sensor Shield 的 S/V/G 排，免麵包板）：
  VCC→5V、GND→G、D0→數位腳（左輪 D2、右輪 D3，這兩支支援中斷，數脈衝最準）

**原理**：盤轉 → 縫切過紅外線 → 一個縫一個「滴答」。20 縫盤轉一圈 = 20 滴答。
數滴答 → 圈數 → 乘輪子周長 = **量出來的**距離。左右輪分開數，轉彎角度也算得出。

**Uno 端要加程式**：數滴答（中斷計數），並把「左 N 滴答、右 M 滴答」隨 `ok` 回報給 Pi。
（`serial_control.ino` 升級，到時我寫。）

## ③ 畫地圖（軟體，②到位後）

Pi 收到左右輪滴答數 → 算出位移與轉角 → 累計成 (x, y, 朝向) → 把走過的格子標記
「已割」——**這就是模擬器裡那張覆蓋圖的真實版**。可以重用 sim.js 的視覺化邏輯，
把真車軌跡畫在網頁上。

## ④ 梯形分解（終點站）

有了地圖 + 位置，把 `sim.js` 的 `initBcd / stepBcd` 移植成 Python：
分區 → 每區來回掃 → 區間接力。演算法邏輯**直接重用**，只是輸入從
「上帝視角的格子陣列」換成「③ 建出來的真實地圖」。

## 常見問題

**Q：為什麼不直接跳到編碼器？**
A：可以！採購到貨前的空檔，① 讓軟體先成形（記錄、座標、畫圖那套），
之後只是把「猜的距離」換成「量的距離」，一行都不浪費。

**Q：循跡模組（TCRT5000）不是可以客串編碼器？**
A：理論可以（反射式數縫），但對位很挑、訊號毛。當免費實驗可玩，正解還是買 ㄇ 字形的。

**Q：誤差累積最後怎麼根治？**
A：那是 SLAM 的課題（用視覺/地標修正位置）。先不煩惱，BCD 在小院子用編碼器
里程計 + 定期回充電座歸零，已經夠用。
